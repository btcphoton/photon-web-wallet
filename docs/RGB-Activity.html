Root Causes (4 layered bugs)

1
CRITICAL: Wrong backend called for RGB transfers

src/utils/rgb-wallet.ts:481–503

fetchRegtestRgbTransfers() calls ${apiBase}/rgb/transfers. The apiBase resolves to the faucet Node.js backend (https://faucet.photonbolt.xyz/api), not the Python thin backend (https://airdrop.photonbolt.xyz).

The faucet's /api/rgb/transfers endpoint (faucet/server.js:4987) is the OLD custodial model. CLAUDE.md explicitly says: "Do not use faucet/server.js for RGB wallet operations." All RGB operations now go through the Python thin backend. The faucet has no visibility into the Python backend's wallet state. 
// rgb-wallet.ts:485-490 — calls FAUCET, not Python thin backend
const { apiBase } = await getRegtestRgbBackend('transfer lookup')
// apiBase = 'https://faucet.photonbolt.xyz/api'  ← WRONG
const response = await fetchWithAuth(`${apiBase}/rgb/transfers`, ...)
2
CRITICAL: listPending() in photon-api.ts is dead code — never called

src/utils/photon-api.ts:132–138

There IS a function that correctly calls the Python thin backend's GET /pending endpoint. But it is imported with an underscore alias and never used:

// photon-api.ts:132
export async function listPending(keys: PhotonKeys): Promise<...> {
  const res = await photonFetch(BASE_URL, 'GET', '/pending', null, keys)
  // ↑ This is correct — calls airdrop.photonbolt.xyz
}

// App.tsx — it's imported but NEVER invoked:
import { ..., listPending as _listPending, ... }
// _listPending is referenced nowhere in App.tsx
3
SIGNIFICANT: Python backend /pending filters out SETTLED transfers

backend/app/routes/balance.py:118–142

Even if listPending() were called, it would only return transfers that are NOT yet settled. Completed transactions (the ones users want to see in history) are excluded:

# balance.py:138
for t in xfers:
    if hasattr(t.status, "name") and t.status.name == "SETTLED":
        continue  # ← SETTLED (confirmed) transfers are SKIPPED
    transfers.append(_pending_transfer(t, asset_id=None))
The Python backend has NO endpoint that returns the full transfer history (settled + pending) for a specific asset. This endpoint needs to be created.
4
SECONDARY: Asset metadata timing guard can silently drop transfers

App.tsx:1907–1910

Even if RGB transfers are fetched correctly, they are silently dropped if the asset hasn't loaded into the assets state yet:

// App.tsx:1906-1910
const assetMeta = assets.find((asset) => asset.id === assetKey)
if (!contractId || !assetMeta) {
  return []  // ← silent empty result if asset not loaded yet
}

This is a timing race: if loadActivities() fires before loadAssetsForNetwork() completes, the asset won't be found and transfers will be silently dropped.

Solution — 3 Steps (fix one by one)

A
Step A: Add GET /transfers endpoint to Python backend

backend/app/routes/balance.py — add new endpoint

Create a new endpoint that returns ALL transfers for a wallet (settled + pending), optionally filtered by asset_id:

# Add to backend/app/routes/balance.py

class TransferRecord(BaseModel):
    idx: int
    kind: str
    status: str
    txid: Optional[str] = None
    asset_id: Optional[str] = None
    amount: Optional[int] = None
    created_at: int
    updated_at: int
    settled_at: Optional[int] = None

class TransfersResponse(BaseModel):
    transfers: List[TransferRecord]

@router.get("/transfers", response_model=TransfersResponse)
async def list_transfers(
    asset_id: Optional[str] = Query(None),
    auth: AuthContext = Depends(require_signed_request),
) -> TransfersResponse:
    wallet, online = manager.get(auth.fingerprint)
    try:
        wallet.refresh(online, asset_id, [], False)
    except rgb_lib.RgbLibError:
        pass
    try:
        xfers = wallet.list_transfers(asset_id)
    except rgb_lib.RgbLibError as err:
        raise HTTPException(status_code=500, detail=str(err))
    records = []
    for t in xfers:
        records.append(TransferRecord(
            idx=t.idx,
            kind=t.kind.name if hasattr(t.kind, "name") else str(t.kind),
            status=t.status.name if hasattr(t.status, "name") else str(t.status),
            txid=getattr(t, 'txid', None),
            asset_id=asset_id,
            amount=getattr(t, 'amount', None),
            created_at=t.created_at,
            updated_at=t.updated_at,
            settled_at=getattr(t, 'settled_at', None),
        ))
    return TransfersResponse(transfers=records)

Step B: Add listTransfers() to photon-api.ts

src/utils/photon-api.ts — add new function

export interface PhotonTransferRecord {
  idx: number
  kind: string
  status: string
  txid: string | null
  asset_id: string | null
  amount: number | null
  created_at: number
  updated_at: number
  settled_at: number | null
}

export async function listTransfers(
  keys: PhotonKeys,
  asset_id?: string
): Promise<{ transfers: PhotonTransferRecord[] }> {
  const qs = asset_id ? `?asset_id=${encodeURIComponent(asset_id)}` : ''
  const res = await photonFetch(BASE_URL, 'GET', `/transfers${qs}`, null, keys)
  if (!res.ok) throw new Error(`Transfers failed: ${await res.text()}`)
  return res.json()
}

Problem 2 — Main Wallet Activities Empty

P2
Main dashboard Activities tab shows "No activities yet" even though wallet has 9.52 BTC balance
App.tsx:4655 — activities.length === 0

Root Causes

1
Only one address is queried for BTC transactions

src/App.tsx:1891 & src/utils/bitcoin-activities.ts:28

fetchBtcActivities(walletAddress, ...) only queries the single walletAddress against the Esplora API. However, the PhotonBolt wallet spreads BTC across multiple addresses:

  • mainBalanceAddress — primary receiving address
  • coloredAddress — RGB-colored UTXO holder
  • utxoHolderAddress — UTXO slot address
  • dustHolderAddress — dust collector

If BTC was received at coloredAddress (e.g., for RGB UTXO allocation fees), it won't appear when only walletAddress is queried.

// bitcoin-activities.ts:28 — only one address queried
const response = await fetch(`${baseUrl}/address/${address}/txs`)
// ↑ Only 'walletAddress' — other wallet addresses are NOT checked
2
Esplora API may not index regtest addresses that receive via RPC

src/utils/backend-config.ts:152

For regtest, the BTC API base is https://faucet.photonbolt.xyz/api. This Esplora instance indexes the regtest chain. However:

  • If the faucet sent BTC using sendtoaddress RPC, the tx should be indexed
  • If there's any Esplora sync lag, recent transactions won't appear
  • On regtest, unconfirmed mempool transactions may not be returned by the API
The wallet shows 9.52 BTC balance — this balance comes from a different source (Electrum/backend balance check), not from Esplora. The balance being non-zero proves the wallet has funds, but Esplora may lag behind or query the wrong address set.
3
Timing: loadActivities useEffect may fire before addresses are populated

App.tsx:1995–2003

// App.tsx:1996-2003
useEffect(() => {
  if (view === 'dashboard' && walletAddress) {
    loadActivities()
  }
}, [view, walletAddress, selectedNetwork])
// 'walletAddress' is the simple wallet address — but mainBalanceAddress,
// coloredAddress etc. are set asynchronously and may not be ready yet.

The effect fires when walletAddress changes, but mainBalanceAddress, coloredAddress, etc. may still be empty. The walletAddresses set used for activity deduplication will be incomplete.

Solution

A
Fetch activities for ALL wallet addresses and merge by txid

src/utils/bitcoin-activities.ts

Modify fetchBtcActivities to accept an array of addresses, fetch each, deduplicate by txid:

export const fetchBtcActivities = async (
    addresses: string[],     // ← was single string, now array
    network: WalletNetwork,
    allWalletAddresses: string[] = []
): Promise<BitcoinActivity[]> => {
    const seen = new Set<string>()
    const all: BitcoinActivity[] = []
    for (const addr of addresses.filter(Boolean)) {
        const acts = await fetchForAddress(addr, network, allWalletAddresses)
        for (const a of acts) {
            if (!seen.has(a.txid ?? '')) {
                seen.add(a.txid ?? '')
                all.push(a)
            }
        }
    }
    return all.sort((a, b) => (b.timestamp ?? 0) - (a.timestamp ?? 0))
}
