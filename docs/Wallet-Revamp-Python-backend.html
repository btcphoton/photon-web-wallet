⚡ PhotonBolt Web Wallet — Python Backend Revamp

Integration plan for photon-web-wallet → Python RGB backend at https://airdrop.photonbolt.xyz · Branch: Python-Backend-Airdrop-RGB

1 Overview & Goals

The existing wallet is a React/TypeScript Chrome extension that connects to a Node.js faucet backend (faucet.photonbolt.xyz) and ICP canisters. This revamp replaces all RGB and wallet-registration API calls with the new Python FastAPI backend (airdrop.photonbolt.xyz), which uses rgb-lib under the hood and provides a clean, authenticated REST API for every wallet operation.

Backend Base URL: https://airdrop.photonbolt.xyz (proxied via nginx to 127.0.0.1:3531)

Auth model: Every protected route requires four signed headers:
X-Photon-Fingerprint · X-Photon-Nonce · X-Photon-Timestamp · X-Photon-Signature

The signature is a secp256k1 ECDSA sig of sha256(METHOD\nPATH\nTIMESTAMP\nNONCE\nsha256(BODY)). The extension already has tiny-secp256k1 for exactly this.

4 Client-Side Cryptography (replacing /_dev/* routes)

The extension must never send the mnemonic or private key to the server. The /_dev/* routes exist only for the test wallet.html and are NOT used in the production extension. All signing happens in the browser.

4.1 — Key Derivation: derivePhotonKeys(mnemonic)

The backend uses rgb-lib's BIP32 HD wallet. To replicate the key derivation client-side, use the existing bitcoin-address.ts infrastructure plus bip32 + tiny-secp256k1. 

// src/utils/photon-keys.ts
import * as bip39 from 'bip39'
import { BIP32Factory } from 'bip32'
import * as ecc from 'tiny-secp256k1'
import { createHash } from 'crypto'          // or use js sha256 lib

const bip32 = BIP32Factory(ecc)

export async function derivePhotonKeys(mnemonic: string) {
  const seed = await bip39.mnemonicToSeed(mnemonic)
  const root = bip32.fromSeed(seed)

  // Vanilla (BIP86 taproot) — m/86'/1'/0'  (coin_type=1 for regtest/testnet)
  const vanillaNode  = root.derivePath("m/86'/1'/0'")
  // Colored (rgb-lib internal) — m/86'/1'/1'
  const coloredNode  = root.derivePath("m/86'/1'/1'")
  // Auth key — m/86'/1'/0'/0/0  (same as signing key)
  const authNode     = root.derivePath("m/86'/1'/0'/0/0")

  return {
    fingerprint:     root.fingerprint.toString('hex').slice(0, 8),
    xpub_vanilla:    vanillaNode.neutered().toBase58(),
    xpub_colored:    coloredNode.neutered().toBase58(),
    auth_pubkey_hex: Buffer.from(authNode.publicKey).toString('hex'),
    // keep private key in memory only — needed for signing
    _authPrivkey:    authNode.privateKey!,
  }
}
⚠️ Coin type: rgb-lib uses coin_type 1 for both regtest and testnet, and 0 for mainnet. Verify by calling /_dev/derive_keys with a known mnemonic and comparing fingerprints before shipping.

4.2 — Building Signed Headers: buildPhotonHeaders()

// src/utils/photon-auth.ts
import * as ecc from 'tiny-secp256k1'

function sha256(data: Uint8Array): Uint8Array {
  // use WebCrypto SubtleCrypto.digest or a pure-JS sha256 lib
  return new Uint8Array(await crypto.subtle.digest('SHA-256', data))
}

export async function buildPhotonHeaders(
  method: string,
  path: string,
  body: string,           // JSON-serialised body, or '' for GET
  fingerprint: string,
  authPrivkey: Uint8Array,
): Promise<Record<string, string>> {
  const nonce     = crypto.getRandomValues(new Uint8Array(16))
                         .reduce((s, b) => s + b.toString(16).padStart(2,'0'), '')
  const timestamp = Math.floor(Date.now() / 1000).toString()
  const bodyHash  = await sha256Hex(body || '')

  // Canonical message: METHOD\nPATH\nTIMESTAMP\nNONCE\nBODY_HASH
  const canonical = [method.toUpperCase(), path, timestamp, nonce, bodyHash].join('\n')
  const msgHash   = await sha256Bytes(canonical)

  const sig = ecc.sign(msgHash, authPrivkey)    // 64-byte compact sig
  const sigHex = Buffer.from(sig).toString('hex')

  return {
    'X-Photon-Fingerprint': fingerprint,
    'X-Photon-Nonce':       nonce,
    'X-Photon-Timestamp':   timestamp,
    'X-Photon-Signature':   sigHex,
    'Content-Type':         'application/json',
  }
}

// Convenience wrapper — mirrors signedFetch from wallet.html
export async function photonFetch(
  baseUrl: string,
  method: string,
  path: string,
  body: object | null,
  keys: { fingerprint: string; _authPrivkey: Uint8Array },
) {
  const bodyStr = body ? JSON.stringify(body) : ''
  const headers = await buildPhotonHeaders(method, path, bodyStr, keys.fingerprint, keys._authPrivkey)
  return fetch(baseUrl + path, {
    method,
    headers,
    body: bodyStr || undefined,
  })
}

4.3 — PSBT Signing: signPhotonPsbt()

The backend's /transfer and /utxos/begin return an unsigned PSBT (base64). The extension must sign it with the wallet's private keys before calling /broadcast or /utxos/end. 

// src/utils/photon-psbt.ts
import * as bitcoin from 'bitcoinjs-lib'
import { BIP32Factory } from 'bip32'
import * as ecc from 'tiny-secp256k1'

bitcoin.initEccLib(ecc)
const bip32 = BIP32Factory(ecc)

export async function signPhotonPsbt(
  unsignedPsbtBase64: string,
  mnemonic: string,
  network: bitcoin.Network = bitcoin.networks.regtest,
): Promise<string> {
  const seed  = await bip39.mnemonicToSeed(mnemonic)
  const root  = bip32.fromSeed(seed, network)
  const psbt  = bitcoin.Psbt.fromBase64(unsignedPsbtBase64)

  // Sign each input with both the vanilla and colored key paths
  // rgb-lib embeds bip32 derivation paths in PSBT inputs — use them
  psbt.data.inputs.forEach((input, i) => {
    (input.bip32Derivation || []).forEach(d => {
      const path    = d.path
      const keyNode = root.derivePath(path)
      try { psbt.signInput(i, keyNode) } catch (_) { /* skip wrong key */ }
    })
  })

  psbt.finalizeAllInputs()
  return psbt.toBase64()
}
⚠️ The exact BIP32 paths used by rgb-lib for colored UTXOs differ from vanilla paths. Validate against the /_dev/sign_psbt route on regtest before removing the dev dependency. The dev route can remain as a server-side fallback for regtest only.

12 UX Notes

  • No "pending" numbers — Never show future - settled as a pending count; it accumulates stale unresolved transfers and misleads users. Show only settled as the real balance, with a grey italic "⏳ syncing" indicator when future > settled.
  • Auto-poll — Start 15s polling whenever any asset shows "⏳ syncing". Stop polling when all assets are settled.
  • Mnemonic persistence — Store mnemonic in chrome.storage.local (same as existing extension pattern). Derive _authPrivkey in memory at unlock; never persist the private key.
  • Registration is idempotent — Call POST /register on every unlock; backend returns 200 or 409 (already registered). Either is success.
  • Asset registry click-to-add — Fetch the public registry, show asset cards, let user tap "Add" which saves the asset_id to local storage and then loads its balance from the backend.
  • RGB invoice expiry — Default 24h (86400s). Show expiry countdown in the invoice display screen. Re-generate if expired.
  • PSBT signing failure fallback — If client-side PSBT signing fails, surface a clear error rather than a silent crash. The PSBT paths must match the wallet's HD derivation.
