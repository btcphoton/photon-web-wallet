Should Fix Before Submission
Action All three icon slots point to the same 128×128 PNG

Manifest declares sizes 16, 48, and 128 but all reference logo.png (128×128). Chrome scales it down, causing blurry toolbar and extension-page icons.

manifest.json : lines 41–44

"icons": {
  "16": "logo-16.png",
  "48": "logo-48.png",
  "128": "logo.png"
}

Create logo-16.png (16×16) and logo-48.png (48×48) in public/ from the master logo asset, then update manifest.

Action 137+ console.log statements in production bundle

Reviewers inspect the packaged source. A large number of debug logs signals unfinished work and can reveal internal logic / key-handling flows to attackers who inspect the extension.

src/App.tsx (137), src/utils/bitcoin-transactions.ts (17), src/utils/rgb.ts (11), and others

Strip all logs at build time by adding one option to vite.config.ts:

build: {
  minify: 'terser',
  terserOptions: {
    compress: {
      drop_console: true,
      drop_debugger: true,
    }
  }
}
Required at Submission Privacy policy must be hosted before submitting

The extension stores sensitive data via chrome.storage (mnemonic, wallet addresses, keys). Chrome Web Store requires a publicly accessible privacy policy URL for any extension that handles personal or financial data.

Privacy policy is not declared in manifest.json — it is entered in the Developer Dashboard during submission. You must host it at an HTTPS URL before submitting.

Suggested URL: https://photonbolt.xyz/privacy

Minimum required content: what data is stored, how it is used, whether it leaves the device, how users can delete it.

Pass Content Security Policy — no unsafe-eval

CSP is script-src 'self' 'wasm-unsafe-eval'; object-src 'self'. wasm-unsafe-eval is explicitly permitted by Chrome's MV3 policy for extensions that require WebAssembly (rgb-lib WASM). unsafe-eval and unsafe-inline are absent — this is correct.

manifest.json : line 47

Pass <all_urls> host permission — justified for Web3 wallets

<all_urls> in host_permissions, content_scripts.matches, and web_accessible_resources.matches is standard practice for all Web3 wallet extensions (MetaMask, Phantom, Uniswap Wallet) because provider.js must be injectable into any dApp the user visits. Chrome Store approves this with the justification below.

Required submission justification: "The extension injects a Bitcoin/RGB provider script into web pages so that decentralised applications can request wallet signatures and transaction data. This requires matching all URLs because dApps can be hosted on any domain."

Submission Tips
Justification for <all_urls>
In the "Permissions justification" field of the Developer Dashboard, paste:

"This Bitcoin/RGB wallet extension injects a provider script (provider.js) into web pages so that decentralised applications hosted on any domain can request wallet addresses, sign transactions, and query balances. This requires content script injection on all URLs, as is standard for Web3 wallet extensions."
Store listing description tip
Mention WASM explicitly: "Uses WebAssembly (rgb-lib) for client-side RGB asset management — no private keys are ever sent to a server." This pre-empts reviewer questions about wasm-unsafe-eval.
Version bump before submission
Update "version" in manifest.json from "1.0.0" to the release version. Chrome Store rejects a new upload with the same version number as a previously submitted package.
Package correctly
Zip the contents of dist/ — not the dist/ folder itself. The manifest.json must be at the root of the zip file, not inside a subdirectory. 
cd dist && zip -r ../photon-bolt-wallet.zip .
